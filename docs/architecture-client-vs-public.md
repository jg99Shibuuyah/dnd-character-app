# Why there's a `client/` folder *and* a `public/` folder

> Captured for later review. Question raised: "Why is there a `client/` folder —
> can't it be resorted or combined into `public/`?" Short answer: they're two
> different *kinds* of thing (build source vs. served static assets), they're
> already combined at serve time, and `public/` is shared with pages that aren't
> part of React — so a straight merge isn't appropriate.

## They're two different kinds of thing

### `client/` — a build-*source* tree (not servable as-is)
- `client/src/*.jsx` — JSX / ES-module source a browser **cannot run directly**;
  Vite must compile it.
- `client/package.json` + `package-lock.json` + `node_modules/` — the front-end's
  **own** dependency tree (React, react-dom, react-router, Vite), separate from the
  server's deps in the root `package.json`.
- `client/vite.config.js`, `client/index.html` — Vite's build config and entry template.
- `client/dist/` — the **compiled output**; this is the only part that actually gets served.

### `public/` — already-servable static files (shipped byte-for-byte)
- `styles.css`
- `resources/builtin/*.js` + `resources/companions.js` — classic (non-module) scripts
  that set up global game data.
- `login.js`

## They're already combined — at serve time, not on disk

The Express server mounts both (`src/app.js`, ~L29–32):

```
express.static(public)       → serves /styles.css, /resources/...
express.static(client/dist)  → serves the compiled React bundle
```

So `/` already stitches `public/` + `client/dist/` together at runtime. The folders
are separate **sources** with different lifecycles: `public/` is hand-written static;
`client/dist/` is generated by the build.

## Why `client/` can't just be folded into `public/`

1. **`public/` is served raw to browsers.** Moving `client/src` (JSX), `node_modules`,
   and `vite.config.js` there would mean serving un-runnable source and the whole
   dependency tree to the public — browsers can't execute JSX.
2. **The browser-ready artifact is `client/dist/`**, and that's already merged with
   `public/` when served. You *could* point Vite's `outDir` at a shared folder, but you
   still need the `client/` source tree (with its own package.json / node_modules) to
   produce it.
3. **`public/`'s assets aren't React-only.** `styles.css` is also linked by the
   server-rendered login/reset pages (`src/views/login.html`, `src/views/reset.html`) —
   which live **outside** React — and `client/index.html` pulls the
   `resources/builtin/*.js` globals by URL. So `public/` is deliberately the **shared**
   static home for both the React app and the non-React pages. It predates React
   (`public/index.html` was the legacy vanilla frontend before the React migration).

## What *could* realistically be tidied (optional, not required)

- **Relocate shared static assets into `client/public/`** (Vite's own static dir) so the
  build copies them, leaving root `public/` holding only the login/reset assets.
  *Caveat:* `resources/builtin/*.js` is still needed by the server-rendered login flow,
  so it can't move entirely into the React build.
- **The "two package.json / two node_modules" split** is the more common source of
  "why two folders?" friction — a standard server-vs-client separation. It can be merged
  into one package, but that mixes back-end and front-end dependencies, usually a net
  negative.

## Bottom line

A conventional **source (`client/`) vs. served-static-assets (`public/`)** split, already
unified at serve time, with `public/` shared by non-React pages. The only worthwhile
cleanup is minor asset relocation — not combining the two.
